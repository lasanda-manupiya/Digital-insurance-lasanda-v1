# TwinRisk AI — Demo (Proof of Concept)

A free, local, browser-based proof of concept for **TwinRisk AI**: a 3D digital twin risk
assessment platform for buildings, infrastructure, and insurance risk evaluation.

The MVP proves the core technical loop:

> Load a 3D model → confirm its real-world scale → place virtual sensors → visualise coverage
> in metres → estimate coverage sufficiency → run risk scenarios → generate recommendations
> with quotation impact.

Everything runs locally. No backend, no accounts, no paid services. Sensors and settings are
stored in browser `localStorage`.

## 3. Installation

Requires Node 18+.

```powershell
cd twinrisk-ai-demo
npm install
npm run generate-model   # only needed if public/models/sample-building.glb is missing
npm run dev
```

Open the URL Vite prints (usually `http://localhost:5173`).

## 5. Where to place your own 3D model

Supported formats: **`.glb`**, **`.gltf`**, and **`.ifc`**.

Put a `.glb` file at `public/models/sample-building.glb`, or simply click
**“Load .glb / .gltf / .ifc”** in the top-left toolbar at runtime (no copying needed — the file
is read in-browser). IFC files exported directly from Revit, ArchiCAD, Tekla, Civil 3D etc.
work without conversion: they are parsed locally by the `web-ifc` WASM parser
(`src/utils/ifcUtils.js`), converted to three.js meshes with their original colours, and
rotated from IFC's Z-up into the viewer's Y-up convention. A sample IFC is bundled at
`public/models/sample-building.ifc`.

Note: the parser binary lives in `public/wasm/` and is copied there from `node_modules`
automatically on `npm install` (`scripts/copy-ifc-wasm.mjs`).

The bundled sample model is generated by `scripts/generate-sample-glb.mjs` — a simple building
(main block, stair tower, plant room annex, rooftop plant, entrance canopy) authored in metres,
so "1 unit = 1 m" is genuinely correct for it.

## 6. How to use the app

1. The model loads and **Model information** shows bounding-box width/height/depth.
2. Confirm scale in **Scale confirmation** (see §7).
3. Click **+ Add sensor**, then click anywhere on the model — a sensor drops at the hit point.
4. Select a sensor (in the list or in 3D) and edit its properties; the coverage sphere resizes
   live as you change the radius.
5. In **Coverage summary**, enter the zone name and required area in m².
6. In **Scenario modelling**, set the assumed flood level and click **Run all scenarios**.
7. **Recommendations** lists actions, data gaps, and quotation impact per scenario.

Everything persists in `localStorage` (`twinrisk.sensors`, `twinrisk.scaleSettings`,
`twinrisk.projectSettings`). “Clear all” removes sensors.

### Saving and reopening a project

Use **Save project** (top toolbar) to download a portable `twinrisk-project-<date>.json` file
containing all sensors, scale settings, zone/coverage settings, and the name of the model it
was built against. **Open project** loads such a file back — on this machine or any other.
The 3D model file itself is *not* embedded (models can be hundreds of MB); if the loaded
project was saved against a different model than the one currently shown, the app tells you
which model file to load so sensor positions line up.

## 7. How scale calibration works

Models can be exported in metres, millimetres, feet, or be unitless, so the app never trusts
the file. A single number is maintained:

```
scaleFactor = metres per model unit
realDistanceMetres  = modelDistanceUnits × scaleFactor
modelDistanceUnits  = realDistanceMetres ÷ scaleFactor
```

Three ways to confirm it (all in `ScaleCalibrationPanel`, maths in `utils/scaleUtils.js`):

1. **Accept detected scale** — assumes 1 unit = 1 m (correct for the sample model).
2. **Manual factor** — e.g. a millimetre-exported model needs `0.001`.
3. **Two-point calibration** — click two points on the model whose real distance you know
   (e.g. a door opening ≈ 0.9 m, or a known elevation). The app measures the model-unit
   distance between the clicks and computes
   `scaleFactor = realDistanceMetres / modelDistanceUnits`.
   Example: 20 model units between clicks, real distance 10 m → factor 0.5 m/unit.

Once confirmed, the Model Info panel shows real-world dimensions, and every coverage radius,
distance, and flood-level check uses the factor.

## 8. How sensor coverage radius works

Sensor radii are stored **in metres** (`coverageRadiusMetres`). When rendered
(`CoverageZone.jsx`), the radius is converted to model units:

```
displayRadiusModelUnits = coverageRadiusMetres / scaleFactor
```

So a 2 m heat sensor draws a 2-unit sphere at factor 1, or a 20-unit sphere at factor 0.1.
General sensors draw a transparent colour-coded sphere; floor-based sensors (flood, water leak,
access route) additionally draw a flat circle with an outline ring. Each sensor type has its
own colour (see `data/sensorTypes.js`); overlapping zones remain readable because the
materials are transparent and don't write depth. Visibility can be toggled per sensor.

## 9. How coverage sufficiency is calculated

`utils/coverageEngine.js` uses simple footprint maths (deliberately ignoring overlap and walls
for the MVP):

```
sensorCoverageArea  = π × r²                       (r in metres)
totalCoverage       = Σ sensorCoverageArea          (filtered by sensor type)
coveragePercentage  = totalCoverage / requiredArea × 100
```

| Coverage % | Status |
|---|---|
| 0–40 | Poor coverage |
| 41–70 | Partial coverage |
| 71–90 | Good coverage |
| 91+ | Strong coverage |

Additional sensors needed = `ceil((requiredArea − totalCoverage) / averageSensorArea)`.
Gap analysis is textual (no room-boundary geometry yet): sensors not linked to the selected
zone, hidden coverage zones, or no sensors of the selected type. Recommendations are
type-specific (e.g. heat sensors → plant rooms, boiler rooms, server rooms).

Worked example (the spec case): Plant Room, required 36 m², two heat sensors at r = 2 m →
2 × 12.57 = 25.13 m² → 69.8 % → **Partial coverage** → "add one more heat sensor…".

## 10. How scenario risk logic works

`utils/riskEngine.js` evaluates five rule-based scenarios; each returns risk level, reason,
affected sensors, coverage status, recommendation, and quotation impact:

| Scenario | Rule |
|---|---|
| **Flood** | No flood/water-leak sensors → **Data Gap**. Any such sensor below the assumed flood level (y × scaleFactor ≤ level) → **High** (low-level exposure zones). Otherwise **Low**. |
| **Fire** | No fire/smoke/heat/temperature sensors → **Data Gap**. Else coverage % vs required area: ≤40 % → **High**, ≤70 % → **Medium**, else **Low**. |
| **Security** | None → **Data Gap**; fewer than 2 security/CCTV sensors → **Medium**; else **Low**. |
| **Structural** | None → **Data Gap**; any structural sensor → **Low** (monitoring available). |
| **Access/evacuation** | None → **Data Gap**; 1 sensor → **Medium**; 2+ → **Low**. |

Quotation impact mapping: High → *may increase*; Medium → *may need review*; Low → *may
decrease or remain stable*; Data Gap → *additional assessment required before quotation
confidence can improve*. This directly supports the business workflow: identified risk raises
the quote; implementing recommendations and re-running scenarios shows the quote can reduce.

## 11. Future expansion notes

- **Deeper IFC / Revit integration** — IFC *geometry* import is already implemented via
  `web-ifc`. Next step: extract BIM metadata through the same API (storeys, `IfcSpace` rooms,
  element properties, project units) to auto-confirm scale and link sensors to real rooms.
- **Accurate BIM room detection** — read `IfcSpace` boundaries to auto-fill required zone area
  and do true point-in-room coverage tests instead of the manual m² input.
- **Report generation / evidence-based PDF** — sensors, coverage, scenario results and
  screenshots (`gl.domElement.toDataURL()`) rendered to PDF with `jspdf` or `react-pdf`,
  including the evidence references already captured per sensor.
- **Real IoT sensor mapping** — give each virtual sensor an external `deviceId`; a FastAPI
  backend ingests MQTT/webhook readings and streams live values into the existing sensor
  objects; status moves `Virtual → Installed`.
- **AI recommendation engine** — the deterministic output of `riskEngine`/`coverageEngine`
  (JSON) is an ideal prompt payload for an LLM to draft underwriter-ready narrative
  recommendations; keep the rules as ground truth, use AI only for language and prioritisation.
- **Insurance quotation logic** — replace the qualitative impact strings with a scoring model:
  building value band (e.g. above/below £12 m) → assessment route (desktop / desktop +
  specialist / full site) → per-scenario risk weights → premium adjustment factor.
- **Cloud database & user accounts** — move `storage.js` behind an API interface (it is already
  the only persistence boundary); swap localStorage for Postgres/Supabase + auth without
  touching components.
- **Real-time alerts** — WebSocket channel pushing threshold breaches from mapped IoT devices
  into the scenario panel, with badge state changes.

## 12. MVP limits (intentional)

No authentication, cloud database, payments, real IoT connections, Revit API, exact room
boundaries, or coverage intersection maths. Coverage is footprint-additive and gaps are
heuristic — this version exists to prove the concept end to end, which it does.
